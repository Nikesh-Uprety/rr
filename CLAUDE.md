## Architecture

### Stack Overview
- **Frontend**: React 19 + Vite + TailwindCSS v4 + Radix UI + Wouter (routing) + TanStack Query + Zustand
- **Backend**: Node.js + Express (monolithic, serves both API and static frontend)
- **Database**: PostgreSQL via Drizzle ORM
- **Auth**: Passport.js (local strategy) with session-based auth and 2FA (OTP)
- **File Uploads**: Local filesystem (configurable UPLOADS_DIR; use persistent volume in production)
- **Testing**: Vitest (unit/integration), Playwright (browser E2E)

### Key Architecture Notes

- **Monolithic server**: Express serves both API (`/api/*`, `/auth/*`) and the built frontend in production. In development, Vite HMR middleware intercepts requests.
- **Path aliases**: `@/*` → `client/src/*`, `@shared/*` → `shared/*`. Configured in `vite.config.ts` and `tsconfig.json`.
- **Database layer**: `storage.ts` provides a SQLite-like abstraction over Drizzle. All queries should use `db` from `server/db.ts` or `storage` methods.
- **Session store**: PostgreSQL via `connect-pg-simple`. Sessions persist across server restarts.
- **Admin RBAC**: Defined in `shared/auth-policy.ts`. Roles: `owner`, `admin`, `manager`, `staff`, `csr`. Pages gated by `requireAdmin` middleware (checks role in `ADMIN_PANEL_ROLES`).
- **2FA**: On first login for internal users (`requires_2fa_setup` flag), OTP setup is enforced. OTP codes stored in DB with expiry; email sent via Nodemailer. Set `E2E_TEST_MODE=1` to bypass email in tests.
- **File uploads**: All uploads go to `UPLOADS_DIR` (default `./uploads` in dev). Served statically at `/uploads`. Subdirectories: `products`, `site-assets/hero`, `site-assets/featured_collection`, `site-assets/new_collection`, `site-assets/collection_page`. In production (Railway), mount a persistent volume.
- **Security**: Rate limiting (in-memory, 5-min cleanup), security headers (STS, CSP, X-Frame-Options, etc.), CORS with origin allowlist in prod (`ALLOWED_ORIGINS`).
- **State management**: React Query for server state, Zustand for client state (cart, UI). Auth state via `useCurrentUser` hook (reads session from `/auth/me`).
- **Component library**: Custom Radix UI components in `client/src/components/ui/`. Follow existing patterns when adding new components.
- **Error handling**: Centralized in `server/errorHandler.ts`. Use `handleApiError(res, err, context)` in route handlers.

## Common Patterns

### Adding a New API Endpoint
1. Define table/insert/update schemas in `shared/schema.ts` if new data model.
2. Implement handler in `server/routes.ts`:
   - Wrap in try/catch and use `handleApiError(res, err, "METHOD /path")`.
   - Use `getQueryParam()` for `req.params`/`req.query` to handle type-safe extraction.
   - Apply middleware: `requireAuth` for logged-in users, `requireAdmin` for admin-only.
3. Add corresponding client function in `client/src/lib/adminApi.ts` or `api.ts`.
4. Use React Query in components (`useQuery`, `useMutation`) with proper query keys.

### Modifying Database Schema
1. Update table in `shared/schema.ts`.
2. Add/update Zod schemas for validation.
3. Run `npm run db:push` to apply changes.
4. Update `storage.ts` queries if needed.
5. Update any affected client components.

### Adding a New Admin Page
1. Create page component in `client/src/pages/admin/YourPage.tsx` (lazy-loaded in `App.tsx`).
2. Add API endpoints in `server/routes.ts` under `/api/admin/*`.
3. Check `shared/auth-policy.ts` to ensure role-based access; add page key to `ADMIN_PAGE_KEYS` if implementing per-page RBAC.
4. Update `AdminLayout.tsx` navigation if needed.
5. Add admin API functions to `client/src/lib/adminApi.ts`.

### Auth & Authorization
- `requireAuth` middleware checks `req.isAuthenticated()`.
- `requireAdmin` checks `req.isAuthenticated()` and user role in `ADMIN_PANEL_ROLES`.
- User object attached to `req.user` (type from `shared/schema.ts`).
- For page-level protection, wrap routes in `ProtectedRoute` (client/src/components/ProtectedRoute.tsx).
- 2FA: After login, if `requires_2fa_setup` is true, OTP verification is required. Check `requiresTwoFactorChallenge()` in auth policy.

## Environment Variables

Required in `.env`:
```
DATABASE_URL=postgresql://...   # PostgreSQL connection (SSL enabled)
SESSION_SECRET=<random-string>  # Min 32 chars for session cookie signing
NODE_ENV=development|production
PORT=5000                       # Preferred port (app will try fallback ports in dev)
ALLOWED_ORIGINS=http://localhost:5000,https://yourdomain.com  # CORS in production only

# Email (SMTP)
SMTP_HOST=...
SMTP_PORT=587
SMTP_USER=...
SMTP_PASS=...

# 2FA
OTP_EXPIRY_MINUTES=10

UPLOADS_DIR=/uploads
```

## Testing Strategy

### Unit Tests (Vitest)
- Location: `tests/unit/**/*.test.ts(x)`
- Configured in `vite.config.ts` with jsdom environment.
- Run: `npm run test:unit`

### E2E Tests (Playwright)
- Location: `tests/e2e/**/*.spec.ts`
- Browser: Firefox
- Projects: `setup`, `storefront`, `admin`, `admin-onboarding`, `admin-smoke`
- Use `E2E_TEST_MODE=1` to bypass email sending.
- Recommended workflow: Start dev server on PORT=5001, then run tests with `PLAYWRIGHT_USE_EXISTING=1 PORT=5001 npx playwright test <spec>`
- Helpers: `tests/e2e/helpers.ts` (page objects), `tests/e2e/db.ts` (direct DB access for test setup/reset).

## Known Constraints & Gotchas

- **Uploads persistence**: Without persistent volume on Railway (or if `UPLOADS_DIR` not writable), uploaded files may be lost on redeploy/restart. Check server logs for upload dir writability warning.
- **Bundle size**: Client bundle includes heavy libraries (jspdf, html2canvas). Consider dynamic imports for admin-only features if needed.
- **2FA testability**: OTP codes are currently exposed in API responses for testing (`/auth/login` response in E2E mode). Ensure this stays behind `E2E_TEST_MODE`.
- **Rate limiting**: In-memory store; not shared across multiple processes. OK for single-process dev; consider Redis in multi-instance production.
- **TypeScript paths**: If imports fail, verify both `vite.config.ts` resolve.alias and `tsconfig.json` paths are in sync.
- **Port conflicts**: Dev server tries up to 20 ports (5000–5019). Production uses only `PORT` (default 5000) and fails if busy.
- **Database SSL**: Neon PostgreSQL requires SSL. Ensure `DATABASE_URL` includes `?sslmode=require` or equivalent. Connection pool in `server/db.ts` uses `ssl: { rejectUnauthorized: false }` by default if SSL detected.

## Conventions

- Prefer existing patterns over introducing new abstractions.
- Keep admin and storefront styling aligned (Tailwind utility classes).
- Use stable selectors for E2E tests (avoid brittle text-based selectors when possible).
- Preserve role-based auth behavior when changing login, onboarding, checkout, or order flows.
- Route handlers: centralized error handling with `handleApiError`, use `getQueryParam` for safe param extraction.
- Avoid new dependencies unless justified. Check `package.json` before adding.
- Components in `client/src/components/ui/` follow Radix UI patterns; extend similarly.
